# Stashly マーケティング戦略

## ポジショニング

### コンセプト
「絵文字リアクションで集める × Canvasに自動整理」

Slack上で散らばった重要メッセージを、絵文字リアクションをトリガーにして自動収集・Canvas化するツール。
リスが木の実を集めて蓄えるように、大切なメッセージを集めて整理する。

### 競合状況

| アプリ | 概要 | 料金 | Stashlyとの差別化 |
|--------|------|------|------------------|
| Reacji Channeler | 絵文字で別チャンネルにメッセージ転送 | 無料（Slack公式） | 転送のみ。整理・まとめ機能なし。Canvas出力なし |
| Reaction Bot | リアクション自動付与・チャンネルリダイレクト | 不明 | リアクション「付与」が主目的。収集・まとめではない |
| BookmarkingBot | メッセージをブックマークしてナレッジベース化 | 不明 | 個人ブックマーク寄り。絵文字ベースの一括収集ではない |
| Slack → Notion連携 | メッセージをNotionDBに送信 | Notion側の料金 | 設定が複雑。非エンジニアには敷居が高い |
| Zapier連携 | リアクションをトリガーに外部ツールへ転送 | $19.99/月〜 | 設定が複雑。Slack内で完結しない |
| Slack AI | チャンネル要約・Canvas下書きAI生成 | $10/ユーザー/月 | 要約であり「絵文字ベースで狙って集める」機能ではない |

**結論: Stashlyの直接競合はほぼ存在しない。「絵文字収集 × Canvas自動整理」の組み合わせはブルーオーシャン。**

## ターゲットユーザー

### プライマリ: マネージャー / チームリーダー
- 開発者（ゲーム開発を含む）のチームリーダーが原体験
- チームの重要な発言をサッと集めてまとめたいニーズ
- ユースケース:
  - 週次レビューで良かったこと（:star:）を集めてCanvas化
  - 意思決定ログの収集（:memo: 付きメッセージ）
  - ふりかえり用の素材集め

### セカンダリ: CS / QA / フィードバック収集
- バグ報告・ユーザーフィードバックに特定の絵文字をつける運用
- QAチームが :bug: リアクションしたメッセージを一覧化
- ゲーム開発のプレイテストフィードバックまとめ

### ターシャリ: エンジニアチーム
- 障害対応ログ、技術メモ収集
- 「自分で作れるけど時間がもったいない」層
- 価格感が合えば刺さる

### 共通する課題
**「チームで働いている人が、Slack上に散らばった情報を整理したい」**
→ 個人利用ではなく、チーム単位の課題解決がコア

## プラン設計

### 課金モデル
- **ワークスペース単位の月額課金**
- チームの課題解決ツールなので、ユーザー単位ではなくワークスペース単位が自然
- チーム人数に関わらず固定料金 → 導入ハードルが低い

### 制限軸
- **B: 収集規模（チャンネル数・メッセージ上限・Canvas追記）をメインの差別化軸とする**
- 課金動機は「もっと便利に使いたい」が自然

### Free プラン

| 項目 | 制限 |
|------|------|
| チャンネル | 実行チャンネルのみ |
| メッセージ上限 | 50件 / 回 |
| 期間 | 過去30日 |
| Canvas | 作成のみ（追記不可） |

### Pro プラン — $5/ワークスペース/月

| 項目 | 内容 |
|------|------|
| チャンネル | 複数チャンネル横断（無制限） |
| メッセージ上限 | 500件 / チャンネル |
| 期間 | 全期間 |
| Canvas | 作成 + 追記 |

### 課金動機の強さ（強い順）
1. **メッセージ上限 50→500** — 本格運用で最初にぶつかる壁
2. **複数チャンネル横断** — チーム運用なら必須
3. **Canvas追記** — 蓄積型の運用ができるかどうか
4. 期間（全期間） — あると嬉しいが単体では課金動機になりにくい

## 年払いディスカウント

- **ローンチ時は月払い（$5/月）のみ**
- 年払いは導入しない。ユーザーが増えて需要が見えてから検討
- 導入する場合の想定: $48/年（月$4相当、20%オフ、2ヶ月分お得）

## 無料トライアル

- **トライアル期間は設けない**
- Freeプランが実質トライアルの役割を果たす
- $5/月は「とりあえず1ヶ月試す」のハードルが十分低い価格帯
- トライアル管理（期限管理、通知、自動ダウングレード等）の実装コストも省ける

## Go-to-Market戦略

### フェーズ分け

| フェーズ | やること | タイミング |
|----------|----------|------------|
| Phase 1 | Slack Marketplace掲載 | OAuth2対応完了後すぐ |
| Phase 2 | Product Hunt ローンチ | Marketplace掲載直後 |
| Phase 3 | SNS・ブログ開始 | 継続的に |
| Phase 4 | コミュニティ展開 | ユーザー事例が出てから |

### チャネル詳細

**Slack Marketplace（最優先・メインの集客チャネル）**
- 「emoji reaction」「canvas」「collect」あたりの検索キーワードで発見される
- 掲載にはOAuth2 + HTTP Mode対応が必須
- レビュー審査があるため、申請→掲載まで数週間かかる想定

**Product Hunt**
- ローンチ時の初速づくりに活用
- 「Slack向けツール」カテゴリは競合少なめでランキング入りしやすい

**SNS / コンテンツマーケ**
- X（Twitter）: 開発者・PMが多い。開発ストーリーを発信して認知獲得
- ブログ記事: 「Slackのメッセージ整理術」等のHow To系で検索流入狙い
- Reddit: r/Slack、r/productivity あたりに投稿

**コミュニティ・直接営業**
- ゲーム開発コミュニティ（原体験に近い層）
- Slack活用系のイベントやコミュニティ

## オリジンストーリーの活かし方

### ゲーム開発チームという原体験

Stashlyはゲーム開発チームでの実体験から生まれたツール。プレイテストのフィードバックがSlackに散らばって整理が大変だった、という課題が原点。

### マーケティングでの活用

| 場面 | 活かし方 |
|------|----------|
| LP | 「Born from a game dev team」的なセクション |
| Product Hunt | ストーリー型の紹介文で差別化 |
| SNS | 開発背景の発信で親近感 |
| Marketplace | 説明文にユースケースとして記載 |

### ゲーム開発向けユースケース例
- :bug: でバグ報告収集 → Canvas化してQAリストに
- :star: でプレイテストの良かった点を集めて振り返り
- :memo: で仕様決定の議論ログを収集

### 注意点
- オリジンストーリーとしては積極的に出す
- ただしプロダクトのポジショニングは **「チームなら誰でも」** を維持
- ゲーム開発はユースケースの1つとして見せる（絞りすぎない）

## Enterpriseプラン

- **現時点では設計しない。需要が見えてから検討する**
- LPやMarketplaceに「Enterprise? Contact us」的な導線だけ置いておく
- 将来的に求められそうな機能の見通し:
  - SSO / SAML認証
  - 監査ログ
  - 複数ワークスペース対応（Enterprise Grid）
  - メッセージ上限の大幅緩和 or 無制限
  - 優先サポート（SLA付き）
  - カスタムインテグレーション（Notion、Confluence等への出力）

## 将来の有料機能候補

現時点では実装しない。ユーザーからの要望を優先し、実装コストが低くて価値が高いものから着手する。

| 機能 | 概要 | 期待値 |
|------|------|--------|
| スケジュール実行 | 毎週金曜に自動で収集→Canvas化 | 高 |
| 複数絵文字同時収集 | :bug: と :warning: を一度に収集 | 高 |
| Canvas以外への出力 | Notion、Google Docs、Markdown export | 中 |
| 収集フィルター | 特定ユーザー、キーワード、期間の細かい絞り込み | 中 |
| Slack Workflow連携 | Workflow Builderのステップとして使える | 中 |
| ダッシュボード | 収集履歴・統計の可視化 | 低 |
| 通知設定 | 収集完了をDMやチャンネルに通知 | 低 |
